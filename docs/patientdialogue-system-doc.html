1. Overview

The current system is not a classic branching dialogue tree. It is an interview state machine.

The core idea is:

question chosen → tone/trust/patience update → channel pressure increases → facts can surface → surfaced facts unlock new questions

This keeps the interview gradual. The player does not see all questions at once. New questions only appear after the patient has said enough to create an opening. If an opening is missed and the first pass ends, that topic can stay missed for the rest of the run.

What the system tracks

Trust and patience are separate.

Trust changes how much the patient volunteers. Patience changes how much more they will tolerate.

What questions do

Questions do not contain the whole answer tree. They mainly apply pressure to one or more channels such as head, general, work, or skin.

What facts do

Facts are the real content units. Each fact has thresholds for becoming hinted or confirmed, plus personality-specific text.

3. Interview loop

  1. The screen opens with Vera’s opener and a personality primer.
  2. The player chooses one opening approach.
  3. The system enters first_pass.
  4. Broad questions are asked. Each one modifies trust/patience and raises channel pressure.
  5. Facts can move from hidden to hinted or confirmed.
  6. Once a fact is at least hinted, more specific questions can become visible.
  7. Ending the first pass unlocks the notebook and blocks all still-hidden facts.
  8. The player selects up to four symptoms.
  9. The player returns once for a follow-up pass, spending one patience.
  10. Only selected symptoms that have an opening can be asked about.
Important current rule: ending the first pass calls blockHiddenFacts(). Any fact that is still hidden becomes blocked and can no longer unlock later in that run.

4. Runtime state

The runtime state is created by createInitialState() in patientdialogue.js.

const state = {
  phase: "opening",
  trust: 0,
  patience: 5,
  askedQuestionIds: [],
  askedQuestionCounts: {},
  askedFollowupFactIds: [],
  channelPressure: {
    general: 0,
    head: 0,
    stomach: 0,
    skin: 0,
    work: 0,
    home: 0,
    function: 0,
    habits: 0
  },
  factStates: { factId: "hidden" | "hinted" | "confirmed" | "blocked" | "exhausted" },
  factDetailStage: { factId: 0 },
  selectedSymptoms: Set(),
  activeFollowupSymptoms: [],
  notebookUnlocked: false,
  isBackWithPatient: false,
  openingQuestionId: null,
  lastQuestionId: null
}

Fact state order

StateOrderMeaning
blocked-1Missed permanently for this run.
hidden0Not surfaced yet.
hinted1Enough has surfaced to create an opening.
confirmed2The interview has made the fact strong enough to stand on its own.
exhausted3Reserved for over-asked facts. Defined now, barely used yet.

7. How to write more

To add a new broad question

  1. Add a new entry in questions.js.
  2. Choose its phase, tone, and channels.
  3. Add visibleWhen rules if it should only appear after an opening exists.
  4. Append its ID to the patient interview’s questionOrder.

To add a new symptom or context fact

  1. Add a new fact in the patient fact file, such as vera-facts.js.
  2. Pick its channels.
  3. Define hinted and confirmed thresholds.
  4. Write response lines for each current personality used by the prototype.
  5. Add a followup block if the notebook should let the player press on it later.
  6. If it is a true symptom the player should be able to select in the notebook, add its symptom label to notebookSymptoms.

To add a new patient after Vera

  1. Create a new fact file, for example facts/martin-facts.js.
  2. Create a new interview blueprint, for example interviews/martin-interview.js.
  3. Reuse the shared personalities and question file where possible.
  4. Only make a new question file if the patient truly needs different question IDs or different visibility logic.

Authoring advice

  • Keep questions abstract and reusable. Put truth-specific text into facts.
  • Use channels to decide what a question is “about.”
  • Use tone to decide trust and patience impact.
  • Use fact thresholds to control when something becomes visible or stays hidden.
  • Use context facts for red herrings and occupation/life-event flavor.
  • Reserve follow-up questions for facts that have already been hinted, unless you intentionally want the player to get a denial.

8. Current limits and known simplifications

  • This is still Vera-only. The architecture is broader than the content.
  • answerStyle is stored on personalities, but is mostly descriptive right now.
  • factDetailStage exists in state, but staged multi-layer fact progression is not fully exploited yet.
  • exhausted exists as a fact state, but the current prototype mostly handles repetition through exhausted response text and patience penalties instead of a rich stage system.
  • Follow-up symptom selection currently only works for symptoms whose fact includes a followup block.
  • The prototype currently uses a fixed Vera personality from the blueprint, not a randomized one.

9. Short version

The MVP works because it separates the system into four layers:

  1. Questions decide pressure and tone.
  2. Personalities decide reaction and thresholds.
  3. Facts decide what truth can surface and how it is phrased.
  4. Interview state decides what is visible now and what has been missed.

That is the part worth preserving as the system grows.
